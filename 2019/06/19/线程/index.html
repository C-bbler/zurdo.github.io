线程

Posted on 2019-06-19

与进程不同的是，同一个进程的线程的地址都是互相共享的。

进程是系统为执行任务而分配的资源，进程有自己独立的地址空间，每个进程都只可以访问自己的内存地址。进程之间互不影响都可以供系统内核单独调用。

线程

线程就是轻量级的进程（LWP），其实在Linux中不区分进程和线程，都会创建一个task_struct进行描述。
区分线程和进程最简单的方法就是：同时一个task_struct访问地址空间的就是进程，如果同时有多个task_struct访问地址空间的就是线程。

线程的资源

线程的资源分为共享资源和私有资源。

  1. 共享资源：可执行的指令、静态数据、进程打开的文件描述符、当前的工作目录、用户ID、用户组ID
  2. 私有资源：线程ID、PC及相关寄存器、堆栈、错误号、优先级、执行状态和周期

线程库

pthread线程库提供创建，回收，结束线程。

线程的创建 - pthread_create

  1. 成功返回0，失败返回错误码
  2. thread线程对象，通过线程对象来创建一个线程
  3. pthread_attr_t *attr是线程的属性，一般采取默认NULL
  4. rountine 线程执行的函数arg传递给rountine的参数，如果不需要就传入NULL
线程的回收 - pthread_join
  1. 成功返回0，失败返回错误码
  2. thread是指定回收某个线程，若该线程未结束，调用线程会一直阻塞直到thread线程结束
  3. retval是接收函数返回的地址。二级指针，定义一个一级指针传入其地址就行。

线程的结束 -pthread_exit

  1. 结束当前线程
  2. 使用exit() 当前进程中的所有线程都会结束

同步

线程间通信因为进程是共享同一进程的地址空间，所以在通信的时候可以通过全局变量交换数据，痛是需要采取同步或者互斥机制。
线程通信-同步

  1. 多个任务按照约定的先后次序相互配合完成一件事情。
  2. 基于信号量进行操作。
  3. 由信号量决定线程是继续运行还是阻塞等待。

    信号量

    代表某一类资源的数量。
    信号量受保护，只能采用三种方式进行访问，
  4. 初始化
  5. P操作（申请操作）
  6. V操作（释放资源）

    Posix 信号量（Linux遵循Posix规范）

    定义了两种信号量：
  7. 无名信号量（基于内存的信号量），主要用于进程中的线程同步
  8. 有名信号量，可以使用在进程和线程之间通信

    P/V操作

    p操作如果申请不到资源就会一直阻塞（P操作只有两种结果，一种是成功申请资源，信号量减一，另外一种就是没有资源一直阻塞，直至有资源释放）
    v操作之后信号量会增加一，若有任务在等待该资源，内核会唤醒等待资源的任务执行P操作。

信号量操作函数

信号量初始化 - sem_init

  1. 信号量创建成功必须进行初始化操作。
  2. 成功返回0，失败返回EOR。
  3. sem_t *sem是指向要进行初始化的信号量对象（首先要定义一个信号量对象）。
  4. pshared 是表明信号量的适用范围（0表示线程间使用，非0表示进程间使用）。
  5. value表示信号量的初值，0代表没有资源。

信号量初始化 - p/v操作

  1. 成功返回0，失败返回EOR。
  2. sem_t *sem是指向要进行初始化的信号量对象。
  3. 需要访问某个资源，仅需要进行P操作，访问资源结束后进行V操作。
  4. 对信号量初始化一定要在创建线程之前。（因为信号量只能被初始化一次，如果先创建了线程，新建线程和主线程的执行顺序不确定，可能线程直接去操作信号量，而此时信号量还没有被初始化）

生产者消费者问题

生产者消费者是一个很典型的同步问题，只有等待生产者制造出某些物品后，消费者才能进行使用。此时生产者就是V操作（释放资源），而消费者就是P操作（申请资源）。

互斥

其中有两个概念，临界资源和临界区。

  1. 临界资源：临界资源通常是一种共享资源，可以被多个线程访问，但是同时只能被一个任务（进程、线程）访问。
  2. 临界区：访问临界资源的代码称为临界区，要实现临界资源的保护，必须将临界区设置为互斥。

互斥机制-mutex互斥锁

互斥锁的作用，只能被一个任务所持有。任何一个任务在访问资源前申请锁，访问结束后释放锁。

互斥锁的初始化 - pthtread_mutex_init

  1. 成功返回0，失败返回EOR

  2. mutex指向初始化的互斥锁对象

  3. attr 互斥锁的属性，一般使用默认属性 故设置为NULL

    申请锁 - pthread_mutex_lock

  4. 成功返回0，失败返回EOR。

  5. mutex指向要初始化的互斥锁对象。

  6. 如果无法申请到锁，当前任务开始阻塞直到获得锁。

释放锁 - pthread_mutex_unlock

  1. 成功返回0，失败返回EOR。
  2. mutex指向要初始化的互斥锁对象。
  3. 执行完临界区要及时释放锁（若没有释放锁，会导致其他申请锁的任务一直被阻塞）。

实例:线程互斥

编译的两种情况：

  1. 不使用互斥锁：gcc -o test test.c -lphread
  2. 使用互斥锁: gcc -o test tet.c -lphread -D_LOCK_ //-D的作用是传递一个宏定义，相当于在程序中使用#define
    结果分析：
  3. 不加锁的情况：主线程执行count++，后将值赋给value1和value2,此时恰巧主线程的时间片使用完毕，另外一个线程开始运行，此时不会出现异常，但是当count++完成后，未来得及赋值的时候，时间片结束，另外一个线程就会打印出来两个数的值，相当于访问临界资源错误。
  4. 加锁的情况：主线程在执行count++时加锁，此时value1和value2的值就不会出现未赋值的情况，因为此时另一个线程申请不到锁所以不会比较两个变量的值。结束的时候进行解锁操作。执行另外一个线程的时候首先就加锁，这样才不会造成临界资源访问异常。最后解锁。
